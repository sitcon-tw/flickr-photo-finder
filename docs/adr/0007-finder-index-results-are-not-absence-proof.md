# ADR 0007: Finder 搜尋結果不可作為照片不存在的判定

## 狀態

Accepted

## 背景

SITCON 行銷、開發、編輯等找圖者通常會直接從公開 GitHub Pages Finder 進入搜尋，不會先閱讀 repo 文件或資料治理說明。

Finder 建立在 Flickr 公開照片與 Google Sheets 照片索引之上。照片標籤、用途、場景、贊助線索與整理狀態可能來自人工整理或機器輔助候選，索引仍可能缺欄位、缺標籤或尚未人工確認。若介面只呈現精準搜尋感，使用者容易把「沒有命中目前索引」誤讀成「Flickr 沒有這類照片」。

## 決策

公開 Finder 的搜尋、篩選與任務模式只能被描述為找圖索引輔助，不可被文件、UI 或輸出格式描述成完整照片存在性判定。

- 空結果應說明「沒有命中目前索引」，不應暗示 Flickr 沒有相關照片。
- 搜尋與結果狀態應提醒標籤可能不完整，特別是待人工確認與未整理照片。
- `curation_status = ai_labeled` 在一般使用者介面應使用「待人工確認」等描述風險的文字，不使用需要專案背景才懂的術語。
- 任務模式只調整排序情境，不隱藏資料，也不代表已完成用途審核。
- 候選清單保持輕量分享體驗；信任邊界優先放在搜尋前、結果狀態與空結果，而不是每種輸出格式都重複提醒。

## 取捨

優點：

- 降低一般找圖者把索引缺漏誤認為照片不存在的風險。
- 讓未整理或待人工確認照片仍可被探索，不因狀態欄位過早排除大量 Flickr 公開照片。
- 把長期信任邊界寫成 ADR，避免研究紀錄、architecture 與 UI 文案各自維護不同說法。

代價：

- 介面需要承擔少量說明文字，避免 Finder 看起來像完整資料庫查詢。
- 維護者新增搜尋、篩選、AI 助手或候選功能時，需要檢查是否又暗示「找不到就是沒有」。

## 替代方案

- 完全不提示索引限制：體驗最輕，但會放大誤判風險。
- 只在 repo 文件說明：對直接從 Pages 進入的使用者無效。
- 在每個候選輸出都加入提醒：可追溯性高，但會破壞跨組分享與聊天討論體驗。
- 只顯示 reviewed / approved 照片：降低信任風險，但會讓 Finder 失去探索大量 Flickr 公開照片的核心價值。

## 重新評估條件

出現以下情況時，可以重新評估提醒位置與文字密度，但仍不應把 Finder 空結果視為 Flickr 照片不存在的證明：

- 照片索引達到明確定義的活動覆蓋率與人工審核完整度。
- GA4、使用者訪談或找圖失敗案例顯示目前提醒不足或過度干擾。
- 未來新增獨立的已審核素材庫，與 Finder 的探索型索引清楚分流。
