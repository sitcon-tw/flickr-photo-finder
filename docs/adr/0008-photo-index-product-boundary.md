# ADR 0008: Photo Finder 是任務導向照片索引

## 狀態

Accepted

## 背景

SITCON Flickr Photo Finder 的核心需求是讓籌備團隊依工作情境找照片，例如社群宣傳、網站橫幅、贊助提案、贊助成果報告、新聞稿、活動回顧、志工招募、設計素材與對外簡報。使用者通常不是從「哪一年哪個相簿」開始，而是從「我需要一張能完成這個溝通任務的照片」開始。

Flickr 仍是照片與相簿來源；Google Sheets 是正式照片索引與人工維護介面；repo 提供 schema、taxonomy、匯入工具、AI 輔助流程、Apps Script 與公開 Finder。專案不應因找圖需求擴張成 Flickr 替代品、原圖保存系統或人物身份索引。

## 決策

Photo Finder 維持為 Flickr 之上的任務導向照片索引層：

- 不保存原圖，也不取代 Flickr 相簿。
- 不把 repo fixture 或 export cache 當成正式照片資料庫。
- 以工作需求、活動脈絡、場景、用途、構圖、贊助線索與使用提醒改善找圖。
- 支援 AI 產生搜尋級 metadata 候選，但不讓 AI 完成人工 review 或批准公開使用。
- 不導入人臉辨識、人物聚類、face embedding、person id 或自動人名標註。

若未來需要找特定公開人物、講者或工作人員照片，應優先使用活動時間、相簿、公開議程資料、Flickr 原頁脈絡、人工確認素材包與明確同意範圍內的公開欄位；這應視為新的資料治理設計，不是讓 AI 從臉部或外貌推斷身份。

## 取捨

優點：

- 專案範圍保持在可維護的照片索引與找圖工作流。
- 避免把公開活動照片轉成可搜尋個人出現紀錄。
- 降低個資、肖像權、誤標、兒少照片與被視為替活動或贊助商背書的風險。
- 讓搜尋品質改善優先落在低風險 metadata，例如場景、用途、構圖、描述與活動脈絡。

代價：

- 找特定人物時不能像個人相簿產品一樣直接用人臉或姓名聚合所有照片。
- 講者、工作人員或來賓素材需要依活動脈絡、議程、相簿與人工素材包收斂。
- 未來若真的需要人物搜尋，必須另行設計權限、同意、修正與刪除流程。

## 替代方案

- 建立人臉辨識或人物聚類：找人效率高，但會把專案推向身份索引與個資治理系統。
- 新增自動人名欄位：不一定使用生物特徵，但仍會讓跨相簿搜尋某人的照片變得容易，風險邊界改變。
- 只依 Flickr 相簿找圖：風險最低，但無法解決社群、網站、贊助與公關等工作需求。

## 重新評估條件

出現以下情況時，應另開 ADR 或 schema 設計討論：

- SITCON 明確需要已授權、可治理的人物素材庫。
- 有正式同意、修正、刪除與權限管理流程。
- 講者或工作人員搜尋無法透過活動脈絡、議程資料、人工素材包與現有欄位滿足。
- 法務、隱私或社群信任邊界有新的組織級規範。
