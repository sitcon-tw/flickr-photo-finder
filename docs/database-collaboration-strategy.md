# 資料庫與協作維護策略

## 目的

這份文件記錄 SITCON Flickr Photo Finder 的資料庫方向。此工具的資料可以對外公開，但必須讓 SITCON 籌備團隊的志工容易維護、交接與共同整理。

這裡的「資料庫」不只指一套正式資料庫軟體，而是指照片索引資料的權威來源、驗證方式、協作流程與未來遷移路徑。

## 團隊協作前提

SITCON 籌備團隊由不同職能的志工共同維護內容。找圖需求可能來自宣傳、行銷、設計、網站、公關、行政、議程或活動紀錄等角色，但不一定每個人都熟悉程式開發或資料庫操作。

因此資料庫設計必須優先滿足幾件事：

- 不把維護門檻綁死在工程背景。
- 讓不同組別能用自己的工作語言找照片。
- 讓資料修改可以被檢查、討論、回溯。
- 讓公開資料與內部判斷界線清楚。
- 讓未來年度或新活動可以接續維護，而不是重做一套。

## MVP 階段資料庫選擇

MVP 階段建議繼續以 repo 內的資料檔作為權威來源：

- `data/photos.csv`: 照片索引主資料。
- `data/tag-taxonomy.json`: 受控標籤與列舉值。
- `data/sponsorship-items.json`: 贊助品項固定版本資料。

這個選擇不是因為 CSV 是最終答案，而是因為它在目前階段有幾個實際優勢：

- 可公開：資料可以直接放在公開 repo，方便引用、檢視與備份。
- 可版本控管：每次欄位、標籤與照片資料變更都有 commit 紀錄。
- 可驗證：`npm run validate:data` 可以擋下格式錯誤、未知標籤與不完整的 reviewed 資料。
- 可轉換：CSV 可以匯入試算表、SQLite、PostgreSQL 或其他後台工具。
- 低耦合：在欄位與工作流程還在驗證時，不需要先承擔正式後台的維護成本。

## 公開資料邊界

目前資料預設可以公開，但仍應避免把不適合公開的內部資訊塞進資料庫。

可以公開的資料包含：

- Flickr 照片 ID、公開連結與縮圖。
- 活動、年份、相簿等公開脈絡。
- 攝影師署名與授權資訊。
- 場景、氛圍、用途、贊助品項、素材包等搜尋索引。
- 公開使用狀態與整理狀態。

應避免放進公開資料的內容包含：

- 對特定人物的負面描述或敏感判斷。
- 未公開的贊助、合作、議程或內部決策資訊。
- 會讓照片主體被不必要識別、標記或推論的細節。
- 只適合內部溝通的評論。

`internal_notes` 雖然名稱包含 internal，但在公開 repo 中仍必須視為公開資料。若未來需要真正的內部備註，應另外建立不公開的資料層，不應混在公開索引裡。

## 志工維護方式

MVP 階段應支援兩種維護路徑：

1. 工程向維護：直接修改 `data/photos.csv`、`data/tag-taxonomy.json`，送 commit 或 pull request。
2. 非工程向維護：把 `data/photos.csv` 匯入試算表整理，再由熟悉 repo 的志工或 agent 協助轉回 CSV 並跑驗證。

無論哪一種路徑，資料進 repo 前都應通過：

```bash
npm run validate:data
```

相簿匯入後的基本流程：

1. 用 `npm run album:add -- <flickr-album-url> --append` 匯入缺少的照片。
2. 在搜尋介面用 `curation_status = unreviewed` 找出待整理照片。
3. 由熟悉活動脈絡的志工補上場景、氛圍、用途、公開使用狀態與品質分數。
4. 行銷相關照片再補 `sponsorship_items` 與 `sponsorship_tags`。
5. 確認資料足夠後，把 `curation_status` 改為 `reviewed` 或 `featured`。
6. 跑資料驗證並 commit。

## 欄位設計原則

欄位應該服務真實工作流程，不應只為了資料完整而增加。

新增欄位前應先確認：

- 這個欄位會被哪個角色拿來搜尋、篩選或判斷？
- 這個欄位是否能由志工穩定判斷？
- 這個欄位是否適合公開？
- 它是所有照片都需要，還是只適合特定素材包或任務？
- 是否可以先用 `collections`、`internal_notes` 或既有標籤觀察需求？

若欄位只對少數情境有用，應先記錄在文件或素材包流程，不急著變成主資料欄位。

## 未來正式資料庫路徑

若資料量、協作人數或查詢需求超過 CSV 能承受的範圍，可以再導入正式資料庫。

可能路徑：

- SQLite：適合仍以 repo 為主、需要更穩定查詢與轉換時使用。
- PostgreSQL：適合已有正式後台、多人線上編輯、權限控管與 API 需求時使用。
- Airtable 或類似工具：適合非工程志工大量維護，但需要接受外部服務與資料同步成本。
- Google Sheets：適合早期協作整理，但需要明確轉換與驗證流程，避免試算表成為無法追蹤的唯一資料源。

不論未來選哪一種，公開資料輸出都應維持穩定格式，讓搜尋介面、文件與其他工具不必跟著後台實作大幅改寫。

## 判斷何時升級

出現以下情況時，再認真評估正式資料庫：

- 多位志工需要同時線上編輯，CSV merge 成本明顯變高。
- 權限需求變複雜，例如公開索引與非公開審核備註必須分層。
- 查詢需求超過靜態前端能合理處理。
- 需要記錄每筆照片的審核歷程、負責人或狀態流轉。
- 需要提供穩定 API 給其他 SITCON 網站或工具使用。

在這些情況出現前，MVP 的重點仍應放在資料品質、標籤是否符合真實需求，以及志工是否能持續維護。

## 目前結論

現階段應把 `data/` 視為公開、可驗證、可版本控管的資料庫。

下一步不是先導入正式資料庫，而是讓這份資料更容易被志工整理：

- 保持 CSV 欄位清楚。
- 用驗證工具保護資料品質。
- 用文件說明欄位判斷方式。
- 用搜尋介面支援待整理與實際找圖。
- 在真實使用後再決定是否需要新的資料層。
