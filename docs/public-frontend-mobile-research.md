# 公開前端手機版代理研究

## 文件狀態

這份文件是 GitHub Pages 前端手機版重設計的代理研究與 owner 評估紀錄。它不是真人使用者訪談，也不是已完成的 usability test。

本文件用來支援 GitHub issue #5「手機體驗極差」的設計決策。實作前應先由 owner 檢查本文件的代理推測，標記採納、修正或否決，再把可採用的需求回填到 #5。

舊文件 `docs/public-frontend-agent-research.md` 是重構前的多角色代理研究快照；本文件只針對目前前端狀態與手機版工作流。

## 方法與限制

本輪以四個找圖核心角色做代理深度訪談：

| 角色 | 手機任務焦點 |
| --- | --- |
| 社群/宣傳 | 快速找到社群貼文、志工招募、活動回顧候選，複製到聊天工具討論。 |
| 網站/視覺設計 | 找網站橫幅、背景素材、社群版型候選，檢查方向、留白、裁切與預覽。 |
| 贊助/行銷 | 找贊助提案與成果佐證候選，依贊助價值與品項收斂並匯出清單。 |
| 新聞稿/對外文件 | 找正式、代表性、可追溯來源的對外候選，保留整理狀態與來源查證線索。 |

代理依據目前 repo 文件與前端程式推演，不代表真人行為。以下結論分成：

- `確認事實`：可由目前 repo 文件或程式碼支持。
- `代理推測`：角色基於任務情境推演出的需求。
- `owner 評估`：待 owner 標記採納、修正或否決。

## 確認事實

- GitHub Pages 前端是公開、唯讀、無登入門檻的搜尋介面，讀取 Google Sheets `photos` 公開 CSV，不寫回 Sheets，也不保存 credential。
- 公開前端的找圖情境包含社群宣傳、網站橫幅、設計素材、贊助提案、贊助成果、新聞稿、對外簡報、志工招募與活動回顧。
- 目前前端已具備任務模式、搜尋、排序、篩選、候選清單、候選複製格式、AI 助手入口、索引概覽、照片卡片操作與 GA4 事件。
- 目前手機 CSS 主要把桌機區塊堆成單欄；候選清單、AI 助手與索引概覽仍在照片結果前方，不是手機專用工作流。
- 現有 filter、task mode、URL key 與 primary filters 由 `data/interface-registry.json` 管理。手機版應重用同一套 finder state 與 interface policy，不新增一套手機專用 filter 規則。
- `curation_status` 與 `public_use_status` 是整理狀態與使用提醒，不是 Flickr 是否公開的判斷。這些欄位可作為排序、篩選或 detail 參考，但 owner 已判斷它們不是手機卡片與候選輸出的高權重資訊。
- 候選清單只存在瀏覽器當下狀態與 URL query，不寫回 Google Sheets。現有複製格式包含 IM 討論版、協作檢查版與純 Flickr URL。

## 跨角色共識

### 手機主任務是建立可討論候選

代理推測：四個角色在手機上都不是要完成正式素材審核或資料整理，而是要快速建立 3 到 10 張可討論候選，再交給文案、設計、贊助、公關、負責人或資料維護者確認。

owner 評估：採納。手機版主要服務快速挑選照片與建立可討論候選，不以手機完成正式素材審核為目標。

### 結果流應優先於完整工作台

代理推測：手機第一層應是任務模式、搜尋、排序或篩選入口、結果摘要、active chips 與照片結果。候選清單、AI 助手、索引概覽和完整 metadata 應降到第二層，不能阻擋第一批照片。

owner 評估：修正採納。一般狀態下結果流優先於候選清單、AI 助手與索引概覽；但使用者載入包含候選清單的連結時，必須清楚明瞭已有被挑選的圖片。

### Filter 應是暫時面板

代理推測：filter 應收進 bottom sheet 或 drawer。主畫面顯示「篩選 N」與 active chips；開啟面板後才顯示任務重點條件與進階條件。各任務的第一層條件仍沿用 `data/interface-registry.json` 的 primary filters。

owner 評估：採納。Filter 採 bottom sheet，並沿用 `data/interface-registry.json` 既有 primary filters，不另建手機專用規則。

### Candidate 應是固定狀態入口

代理推測：候選清單應改成固定入口，例如「候選 N」，點開 bottom sheet 或 modal 才管理縮圖、移除、清空與複製格式。預設複製格式應偏 IM 討論版，方便在聊天工具中回覆「第幾張」。

owner 評估：採納。Candidate 採固定「候選 N」入口與 sheet/modal 管理。

### Finder 內 preview/detail 是手機 P0

代理推測：設計與對外文件角色需要在不離開 Finder 的情況下檢視完整圖片比例、方向、留白、安全裁切、整理狀態、使用提醒與主要來源操作。手機點照片應先開 Finder 內 preview/detail，而不是直接離開到 Flickr。

owner 評估：採納。Finder 內 preview/detail 納入 #5 P0。

## 角色需求摘要

### 社群/宣傳

代理推測：

- 常用語言偏向「青春感」、「友善」、「交流」、「可放字」、「不要太硬」。
- 手機常見情境是臨時討論、通勤或會議中湊出 5 到 10 張候選。
- 社群任務第一層 filter 應優先保留氛圍、場景、方向、留白與安全裁切；使用提醒不作為手機第一層重點。
- 候選複製預設應短，包含編號與 Flickr URL；完整 Finder、Sheets 與狀態資訊若仍需要，應留在次要格式或 detail，而不是手機預設輸出。

owner 評估：部分採納。採納社群手機版以挑選候選與聊天討論為主；修正為狀態提醒不是高權重資訊，不應進入手機第一層或預設複製內容。

### 網站/視覺設計

代理推測：

- 手機不是正式設計環境，而是建立網站橫幅、背景素材或社群版型候選。
- 卡片第一層應顯示方向、留白與安全裁切；使用提醒不作為手機第一層重點。
- 目前 4:3 cover 縮圖不足以判斷原始構圖；preview/detail 需要看完整圖片比例。
- 大圖、原圖頁、Flickr、Sheets、複製連結要分主次；手機主 action 應是「加入候選」與「預覽」。

owner 評估：採納 preview/detail 與構圖判斷需求；修正為狀態提醒不是手機第一層高權重資訊。

### 贊助/行銷

代理推測：

- 贊助/行銷手機任務是快速找能證明贊助價值的畫面，建立 3 到 10 張可討論候選。
- `sponsorship_tags` 適合作為第一步粗收斂，例如品牌露出、會眾互動、成果佐證。
- `sponsorship_items` 適合作為第二步精準指定品項，必須保留片段搜尋與多選，不應退回只能選完整品項名稱的長 select。
- 協作檢查版候選輸出可包含 Finder 清單、Sheets 連結與 Flickr URL；整理狀態與使用提醒不作為手機候選輸出的必要重點。

owner 評估：採納贊助價值與品項的收斂方式；修正為狀態提醒不是手機候選輸出的必要重點。

### 新聞稿/對外文件

代理推測：

- 手機任務是快速找正式、清楚、有活動識別的代表性照片候選。
- Source traceability 是核心需求：候選至少要追到 Flickr URL、Finder deep link、Sheets row、photo id、活動與相簿脈絡。
- `needs_review`、`avoid`、`ai_labeled`、`unreviewed` 不作為手機卡片與候選輸出的高權重資訊；若需要查證，應留在 detail 或既有排序/篩選語意中。

owner 評估：部分採納。採納來源追溯需求；否決把狀態提醒做成手機卡片與候選輸出的重點。

## 回填 #5 的 P0 範圍

#5 第一輪實作範圍採用：

- 手機主畫面重排：結果流優先，候選清單、AI 助手與索引概覽不阻擋第一批照片。
- 候選連結例外：載入 URL 已包含候選照片時，主畫面需清楚提示目前已有被挑選圖片，並提供進入候選清單的入口。
- Filter bottom sheet：沿用既有 finder state、URL state 與 interface registry，不新增手機專用資料契約。
- Candidate fixed entry + sheet：顯示候選數，支援縮圖、移除、清空與三種複製格式。
- Finder 內 preview/detail：可看完整圖片比例、構圖資訊與主要來源操作；狀態資訊可保留在 detail 中，但不作為手機第一層重點。
- Active chips 水平捲動，支援單一條件移除。
- 主要觸控目標至少 44px。

## 建議驗收

- `pnpm finder:test`
- `pnpm finder:build`
- `pnpm finder:check`
- 以 360x800、390x844、430x932 viewport 人工驗收：
  - 搜尋與切換任務模式。
  - 開啟 filter sheet，套用與移除條件。
  - 開啟 preview/detail，確認可回到 Finder。
  - 加入 3 到 10 張候選。
  - 開啟候選 sheet，移除、清空、複製 IM 討論版與協作檢查版。
  - 以含候選照片的 URL 進入時，能清楚看到已有候選並進入候選清單。
  - 頁面不需要水平捲動，操作區不遮住主要內容。
