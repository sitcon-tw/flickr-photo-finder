# SITCON Photo Finder MVP 記錄

## 背景

SITCON 在 Flickr 上累積了大量對外釋出的活動照片。對籌備團隊來說，找照片常常不是為了瀏覽或回憶，而是為了完成具體工作：社群宣傳、網站視覺、贊助提案、活動回顧、講者素材、志工招募、媒體公關、設計製作等。

這些工作情境的共同痛點是：需求通常不是從「哪一年、哪個相簿」開始，而是從「我現在需要一張能傳達某種感覺、適合某種用途的照片」開始。

因此這個專案的核心不應只是管理照片，而是降低籌備團隊從模糊需求找到可用素材的時間。

## 核心理解

照片搜尋需要同時支援兩種需求：

- 精確脈絡：特定活動、年份、相簿、講者、場景。
- 情緒與用途：熱鬧、專注、溫暖、青春感、專業、幕後感、適合招募、適合贊助簡報、適合網站 hero。

尤其在社群媒體宣傳中，照片常常不是用來精確描述某一場活動，而是用來傳遞受眾會先感受到的情緒，例如社群活力、參與感、友善、歸屬、交流、投入與專業度。

所以第一版應該把「用途」和「氛圍」當作一等公民，而不是只依賴 Flickr 既有的活動相簿結構。

## MVP 目標

MVP 應該先建立一個 Flickr 之上的照片索引層，而不是取代 Flickr 或搬移所有原圖。

第一版目標：

- 匯入或紀錄 Flickr 照片基本資訊。
- 支援人工整理、素材包與標註。
- 支援依用途、情緒、場景與構圖找照片。
- 支援行銷組依贊助品項與贊助價值找照片。
- 保留 Flickr 原始連結、授權與 credit 資訊。
- 能建立素材包，讓常見需求不必每次重新搜尋。

第一版不需要完整整理所有照片。比較務實的做法是先從使用者提供的 Flickr 相簿、近幾年常用活動、常見宣傳情境，或各組已經反覆使用的代表照片建立第一批公開索引。

## MVP 欄位

以下欄位是第一版建議優先支援的最小集合：

| 欄位 | 用途 |
| --- | --- |
| `photo_id` | Flickr 照片唯一識別，避免重複匯入。 |
| `photo_url` | 回到 Flickr 原始頁面，確認脈絡與取得照片。 |
| `image_preview_url` | 在搜尋介面顯示縮圖。 |
| `album_title` | 保留原始相簿脈絡。 |
| `event_name` | 標示活動名稱，例如年會、Camp、Hackathon。 |
| `event_year` | 支援跨年度搜尋與回顧。 |
| `photographer` | 攝影者與署名資訊。 |
| `license` | 判斷是否能對外使用與如何署名。 |
| `scene_tags` | 描述畫面內容，例如講者、會眾、報到、攤位、工作人員、合照、交流、舞台。 |
| `mood_tags` | 描述情緒與氛圍，例如熱鬧、專注、溫暖、青春感、專業、幕後感。 |
| `recommended_uses` | 建議用途，例如社群貼文、網站 hero、志工招募、投稿宣傳、贊助提案、新聞稿、活動回顧。 |
| `sponsorship_items` | 標示照片對應的具體贊助品項。這是行銷組最常見的找圖需求之一，應以 SITCON 贊助徵求書的品項資料為主要來源，而不是用人工概略分類取代。 |
| `sponsorship_tags` | 標示照片能支援的贊助價值或證明力，例如攤位曝光、品牌露出、會眾互動、實體導流、線上曝光、參與者體驗、贊助成果佐證。 |
| `orientation` | 橫式、直式或方形，幫助設計與社群版型選圖。 |
| `has_negative_space` | 是否有留白可放文字。 |
| `safe_crop` | 是否適合裁切成常見比例，例如 1:1、16:9、9:16。 |
| `public_use_status` | 可公開、需確認、避免公開。 |
| `priority_level` | 推薦使用優先度，方便排序與快速挑選候選照片。 |
| `collections` | 素材包關聯，例如志工招募精選、贊助提案、SITCON 介紹。 |
| `internal_notes` | 可公開閱讀的整理脈絡、注意事項或使用建議。 |
| `curation_status` | 未整理、AI 初標、已審核。只描述資料是否經過人工確認；精選或不推薦使用由其他欄位表達。 |

這些欄位的目的不是完整描述照片，而是支援「能不能快速找到並放心使用」。

## 贊助欄位與場景欄位的差異

行銷組常見需求不是只找「有攤位」、「有背板」或「有會眾」的照片，而是找能證明某個贊助品項有價值、能放進贊助提案或成果報告的照片。因此 `sponsorship_items` 和 `sponsorship_tags` 不應該只塞進 `scene_tags`。

三者差異如下：

| 欄位 | 回答的問題 | 例子 |
| --- | --- | --- |
| `scene_tags` | 照片裡看到了什麼場景、人物或活動。 | 攤位、會眾、交流、舞台、報到、茶點、背板。 |
| `sponsorship_items` | 這張照片對應到哪個具體贊助品項。 | 會場攤位、獨家議程、午餐旗與點心旗、迎賓袋文宣、Badge 繩廠商 Logo 曝光、網頁 Logo 曝光。 |
| `sponsorship_tags` | 這張照片能證明或支援哪種贊助價值。 | 品牌露出、會眾互動、實體導流、線上曝光、主舞台曝光、參與者體驗、贊助成果佐證。 |

簡化來說：

```text
scene_tags = visual facts
sponsorship_items = sponsor inventory
sponsorship_tags = sponsor value / proof use
```

例如一張會眾在贊助攤位前交流、背景有贊助商 logo 的照片：

- `scene_tags`: 攤位、會眾、交流、展示桌。
- `sponsorship_items`: 會場攤位。
- `sponsorship_tags`: 攤位曝光、會眾互動、實體導流、贊助成果佐證。

這樣拆分的原因是：同一個場景可能支援不同贊助價值，而同一個贊助品項也可能出現在不同場景。若只用 `scene_tags`，行銷組仍然需要逐張判斷照片能不能支援招商或成果回報；若把 `sponsorship_items` 與 `sponsorship_tags` 獨立出來，才能直接支援「我要找某個贊助品項的實際照片」這種高頻需求。

`sponsorship_items` 的字彙應優先對齊 SITCON 2026 贊助徵求書 `src/data/item.json` 的固定版本資料。該資料已經有精確品項名稱、分類、子品項，以及「人才招募」、「品牌曝光」、「產品推廣」三種用途描述；照片索引不應重新發明另一套不一致的品項名稱。未來年度若有新的贊助徵求資料，應明確建立新的版本資料，視情況取代或延伸 2026 版本。

## 未來可探索欄位

以下欄位保留作為未來想像與評估素材，不應在沒有實際回饋前被視為必做規格。

### 來源與授權

- `download_url`
- `original_size_url`
- `album_id`
- `taken_at`
- `uploaded_at`
- `credit_required`
- `credit_format`

這些欄位可以改善從搜尋到實際取用的流程，尤其是需要下載高解析度照片、產出媒體素材包或統一署名格式時。

### 活動與議程脈絡

- `event_type`
- `venue`
- `session_title`
- `speaker_names`
- `organizer_group`

這些欄位適合在講者宣傳、議程整理、年度回顧與特定場地需求明確增加時再導入。

### 畫面內容

- `people_count`
- `subject_tags`
- `activity_tags`
- `visible_branding`
- `indoor_outdoor`
- `face_prominence`

這些欄位能提升內容搜尋精準度，也有助於判斷人物露出與品牌露出，但標註成本較高，應先確認使用頻率。

### 情緒與社群感

- `energy_level`
- `community_feel`
- `formality`
- `story_notes`

這些欄位可以支援更細緻的社群宣傳判斷，例如找一張「熱鬧但不混亂」、「專業但不疏離」、「有交流感但不過度私密」的照片。

### 設計可用性

- `aspect_ratio`
- `visual_clutter`
- `sharpness_quality`
- `brightness`
- `color_tone`
- `dominant_colors`

這些欄位對設計工作很有幫助，但可能需要實際觀察設計、社群與網站使用者如何找圖後，再決定是否系統化。

### 使用風險與審核

- `sensitive_notes`
- `approved_by`
- `approved_at`
- `not_recommended_uses`
- `audience_fit`

這些欄位適合在照片開始被頻繁用於新聞稿、贊助提案、官網或其他高曝光情境後導入。

### 策展與維護

- `last_reviewed_at`
- `created_at`
- `updated_at`
- `reviewer_notes`
- `source_import_batch`

這些欄位有助於長期維護資料品質，但第一版可以等資料量與協作人數增加後再補。

## 重要原則

這份紀錄不是最終資料模型，也不應被未來視為唯一依據。

實作後必須持續回頭檢查：

- 使用者實際會怎麼描述找圖需求。
- 哪些欄位真的被拿來搜尋或篩選。
- 哪些欄位標註成本高但幫助有限。
- 哪些常見需求仍然找不到照片。
- 哪些照片雖然找得到，但無法放心使用。
- 哪些素材包能大幅減少重複工作。

資料欄位應該跟著實際使用回饋調整。若某些欄位沒有人用，應該簡化；若使用者反覆用自然語言提出某種需求，應該把它轉成更明確的標籤、欄位或素材包。

專案成功的判準不是資料庫多完整，而是籌備團隊能不能更快從「我需要一張適合這個目的的照片」走到「這張可以用，知道怎麼用，也知道去哪裡拿」。
