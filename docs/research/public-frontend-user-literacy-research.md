# 公開前端使用素養代理研究

## 文件狀態

這份文件記錄 2026-05-22 針對 GitHub Pages Finder 使用素養風險的代理訪談結論。它不是真人使用者訪談，也不是完整 usability test；用途是保存本輪 Pages 文案與心智模型改進的研究證據。長期信任邊界以 `docs/adr/0007-finder-index-results-are-not-absence-proof.md` 為準。

本輪背景是：行銷、開發、編輯等找圖者通常只會從公開網址進入 Finder，不會先閱讀 repo README 或維護文件。

## 驗證問題

回饋指出，一般找圖者在使用搜尋與篩選時，可能不知道照片標籤來自不完整索引與機器輔助整理結果，因此容易把「篩選不到」誤讀成「沒有這類照片」。本輪代理訪談檢查：

- 使用者是否知道 Finder 是索引輔助，不是完整判定。
- 使用者是否知道篩選沒有結果不代表 Flickr 沒有照片。
- 使用者是否理解尚未人工確認的照片仍可作為候選，但標籤可能漏掉。
- 風險提示應放在哪些操作節點，才不會變成被忽略的說明文件。

## 代理角色

| 角色 | 模擬任務 | 主要風險 |
| --- | --- | --- |
| 行銷新志工 | 找 5 到 10 張贊助成果報告候選照片 | 把贊助價值、品項與場景篩選當成完整資料庫查詢。 |
| 開發新志工 | 找網站首頁或背景素材 | 把橫式、留白、安全裁切與用途篩選誤讀成不存在判定。 |
| 編輯新志工 | 找活動回顧或新聞稿代表照片 | 把空結果和整理狀態誤讀成素材不存在或只有已整理照片可靠。 |

## 共同觀察

- 風險成立。三個角色都會自然把搜尋與篩選器理解成精準資料庫查詢。
- 「任務模式只調整推薦排序，不會排除照片」有幫助，但只解釋任務模式，不足以說明篩選欄位可能缺標。
- 「AI 標記」對未接觸專案的人不是好理解的公開 UI 詞彙。公開找圖介面應優先使用「待人工確認」這類直接描述使用風險的文字。
- 空結果是最容易形成錯誤結論的時刻。文案應直接說明沒有命中索引不代表 Flickr 沒有相關照片。
- 候選清單分享應保持輕量；信任邊界應主要放在搜尋前、結果狀態與空結果，而不是塞進每次分享輸出。

## 本輪採納

- Pages 一般使用者介面將 `curation_status = ai_labeled` 顯示為「待人工確認」，raw value 仍維持 `ai_labeled`。
- 搜尋區加入短提醒，說明 Finder 是找圖索引，不是完整判定。
- 結果狀態列補充索引限制，提醒待人工確認與未整理照片可能缺少標籤。
- 零結果文案改為操作引導，避免讓使用者推論 Flickr 沒有相關照片。
- 候選清單輸出不加入額外提醒，避免破壞跨組分享體驗。
